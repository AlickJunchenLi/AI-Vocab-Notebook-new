AI Vocabulary Notebook - database.py Function Handoff
Prepared for: Senior Software Developer Partner
Purpose: Explain the expected behavior, parameters, use cases, and implementation logic for the core database.py functions.

============================================================
0. OVERALL CONTEXT
============================================================

The project is an AI Vocabulary Notebook. The database.py file should act as the data layer of the app.

In simple terms:

    main.py / UI layer collects user input
    database.py handles SQLite operations
    SQLite stores vocabulary entries and word relations

The database.py file should NOT control the terminal menu or UI flow. It should provide clean reusable functions that other files can call.

Core tables expected for the MVP:

1. entries_v2
   Stores vocabulary entries.

   Suggested columns:
       id INTEGER PRIMARY KEY AUTOINCREMENT
       language TEXT NOT NULL
       word TEXT NOT NULL
       translation TEXT
       notes TEXT
       created_at TEXT DEFAULT CURRENT_TIMESTAMP
       updated_at TEXT DEFAULT CURRENT_TIMESTAMP

   Suggested uniqueness rule:
       UNIQUE(language, word)

2. relations_v2
   Stores relationships between vocabulary entries.

   Suggested columns:
       id INTEGER PRIMARY KEY AUTOINCREMENT
       source_entry_id INTEGER NOT NULL
       target_entry_id INTEGER NOT NULL
       relation_type TEXT NOT NULL
       created_at TEXT DEFAULT CURRENT_TIMESTAMP

   Suggested uniqueness rule:
       UNIQUE(source_entry_id, target_entry_id, relation_type)

   Suggested foreign keys:
       source_entry_id references entries_v2(id)
       target_entry_id references entries_v2(id)

3. records_v2
   Optional for later. It can store learning history, notes, examples, or review records.

============================================================
1. GENERAL DESIGN PRINCIPLES
============================================================

1. Keep database.py independent from UI.
   It should return useful values instead of only printing.

2. Prefer structured return values.
   For example, return dictionaries, lists of dictionaries, True/False, or clear success/error objects.

3. Avoid duplicate data.
   Use entry_exists(), get_entry_id(), and relation_exists() before inserting.

4. Use SQL placeholders for values.
   Example:
       cur.execute("SELECT * FROM entries_v2 WHERE language = ? AND word = ?", (language, word))

5. Use a whitelist for dynamic column names.
   SQL placeholders cannot safely replace column names, so update_entry_v2() must validate the column name first.

6. Normalize user input before storing or searching.
   For example:
       " Happy " -> "happy"
       " 开心 " -> "开心"

7. Close database connections safely.
   Use context managers where possible:
       with get_connection() as conn:
           ...

2.1 get_connection()
------------------------------------------------------------

Function signature:

    get_connection()

What this function is:

    A helper function that opens and returns a SQLite database connection.

What it is used for:

    Every database operation needs a connection. Instead of writing sqlite3.connect("notebook.db") in every function, all functions should call get_connection().

Parameters:

    None in the MVP version.

Optional future parameter:

    db_path: str or Path
        Allows testing with a different database file.

How it works:

    1. Import sqlite3.
    2. Connect to notebook.db.
    3. Optionally enable foreign keys with PRAGMA foreign_keys = ON.
    4. Optionally set row_factory so rows can be converted to dictionaries.
    5. Return the connection object.

Expected return:

    sqlite3.Connection

Example use:

    conn = get_connection()
    cur = conn.cursor()

Implementation notes:

    If relations_v2 uses foreign keys, this function should enable:

        PRAGMA foreign_keys = ON;

    This makes SQLite enforce relation integrity.

2.3 split_words(text)
------------------------------------------------------------

Function signature:

    split_words(text)

What this function is:

    A helper function that converts comma-separated text into a clean Python list.

What it is used for:

    It helps handle fields where the user may type several words at once, such as synonyms or tags.

Parameters:

    text: str
        The raw input string from the user.

Example input:

    "happy, joyful，cheerful"

Expected output:

    ["happy", "joyful", "cheerful"]

How it works:

    1. If text is None or empty, return an empty list.
    2. Replace Chinese commas with English commas.
       "，" -> ","
    3. Split the text by English comma.
    4. Strip spaces around each item.
    5. Remove empty items.
    6. Return the clean list.

Expected return:

    list[str]

Use case:

    User types:
        happy, joyful，cheerful

    The function returns:
        ["happy", "joyful", "cheerful"]

Implementation notes:

    Python str.split() only accepts one separator at a time. This is why we should first replace "，" with "," and then call split(",").

2.4 normalize_word(word)
------------------------------------------------------------

Function signature:

    normalize_word(word)

What this function is:

    A helper function that cleans a word before saving, searching, comparing, or linking it.

What it is used for:

    It makes duplicate checking and searching more reliable.

Parameters:

    word: str
        The raw word input from the user.

How it works:

    1. If word is None, return an empty string or None depending on chosen design.
    2. Strip spaces from the beginning and end.
    3. For English words, optionally convert to lowercase.
    4. For Chinese words, keep the characters unchanged.
    5. Return the cleaned word.

Expected return:

    str

Example:

    normalize_word(" Happy ") -> "happy"
    normalize_word(" 开心 ") -> "开心"

Use case:

    Without normalization, the database may treat these as different entries:
        "Happy"
        " happy"
        "happy "
        "happy"

    Normalization helps make all of them become:
        "happy"

Implementation notes:

    Be careful with lowercasing all languages. For MVP, a simple design is acceptable:
        word.strip().lower()

    But for a more precise version, only lowercase alphabetic English text.

2.5 entry_exists(language, word)
------------------------------------------------------------

Function signature:

    entry_exists(language, word)

What this function is:

    A helper function that checks whether a vocabulary entry already exists.

What it is used for:

    It prevents duplicate entries.

Parameters:

    language: str
        The language of the entry, such as "English" or "Chinese".

    word: str
        The vocabulary word, such as "happy" or "开心".

How it works:

    1. Normalize the word.
    2. Clean the language.
    3. Query entries_v2 for a row matching language and word.
    4. If a row exists, return True.
    5. Otherwise, return False.

Expected return:

    bool

Example:

    entry_exists("English", "happy") -> True
    entry_exists("English", "unknownword") -> False

Use case:

    add_entry_v2() should call entry_exists() before inserting a new row.

Implementation notes:

    This function should not print too much. It should return True or False so other functions can decide what to do.

------------------------------------------------------------

Function signature:

    get_entry_id(language, word)

What this function is:

    A helper function that returns the database id for a specific entry.

What it is used for:

    relations_v2 should link entries by id, not by raw text. Therefore relation functions need this helper.

Parameters:

    language: str
        The language of the entry.

    word: str
        The vocabulary word.

How it works:

    1. Normalize the word.
    2. Clean the language.
    3. Query entries_v2 where language and word match.
    4. If found, return the id.
    5. If not found, return None.

Expected return:

    int | None

Example:

    get_entry_id("English", "happy") -> 3
    get_entry_id("English", "unknownword") -> None

Use case:

    add_relation_v2() needs source_entry_id and target_entry_id.

Implementation notes:

    Do not create a new entry inside this function. This function should only find an existing id.

2.7 add_entry_v2(language, word, translation=None, notes=None)
------------------------------------------------------------

Function signature:

    add_entry_v2(language, word, translation=None, notes=None)

What this function is:

    A CRUD Create function. It adds a new vocabulary entry to entries_v2.

What it is used for:

    When the user wants to save a new word in the notebook.

Parameters:

    language: str
        Required. The language of the word.

    word: str
        Required. The vocabulary word.

    translation: str | None
        Optional. A simple translation or meaning.

    notes: str | None
        Optional. User notes, usage notes, or explanation.

How it works:

    1. Clean language.
    2. Normalize word.
    3. Validate that language and word are not empty.
    4. Call entry_exists(language, word).
    5. If the entry already exists, return a duplicate message or result.
    6. Insert the new row into entries_v2.
    7. Commit the transaction.
    8. Return success information.

Expected return:

    Recommended structured result:

        {
            "ok": True,
            "message": "Entry added successfully.",
            "entry_id": 3
        }

    For duplicate:

        {
            "ok": False,
            "error": "Entry already exists."
        }

Use case:

    add_entry_v2("English", "happy", "开心", "positive emotion")

Implementation notes:

    The database should also have UNIQUE(language, word) as a second layer of duplicate protection.

2.10 update_entry_v2(language, word, column, new_value, operation="replace")

What this function is:

    A CRUD Update function. It updates an existing vocabulary entry.

    It supports three update modes:

        replace
            Replace the whole value of a column.

        add
            Add one value into a list-style column.

        delete
            Delete one value from a list-style column.

What it is used for:

    It allows the user to edit normal entry fields, and also manage list-style synonym/translation fields.

Parameters:

    language: str
        The current language of the entry to find.

    word: str
        The current word of the entry to find.

    column: str
        The column to update.

    new_value: str
        The new value to replace, add, or delete.

    operation: str
        The update mode.

        Allowed values:
            "replace"
            "add"
            "delete"

Allowed columns:

    language
    word
    synonym
    translation
    notes

List-style columns:

    synonym
    translation

Rules:

    1. operation="replace" can be used for:
           language
           word
           synonym
           translation
           notes

    2. operation="add" can only be used for:
           synonym
           translation

    3. operation="delete" can only be used for:
           synonym
           translation

How it works for replace:

    1. Validate the column name using a whitelist.
    2. Find the entry by current language and current word.
    3. If the column is synonym or translation, convert new_value into a clean list using split_words().
    4. Store the list as JSON text using json.dumps(..., ensure_ascii=False).
    5. If the column is word, normalize the new word.
    6. Run UPDATE entries_v2 SET column = ? WHERE language = ? AND word = ?.
    7. Update updated_at.
    8. Commit and return success/failure.

How it works for add:

    1. Only allow column to be synonym or translation.
    2. Find the entry.
    3. Read the current JSON list text from the column.
    4. Convert it into a Python list using split_words().
    5. If new_value already exists in the list, return duplicate result.
    6. Otherwise append new_value.
    7. Convert the updated list back to JSON text.
    8. Save it into SQLite.
    9. Commit and return success/failure.

How it works for delete:

    1. Only allow column to be synonym or translation.
    2. Find the entry.
    3. Read the current JSON list text from the column.
    4. Convert it into a Python list using split_words().
    5. If new_value does not exist in the list, return not-found result.
    6. Otherwise remove new_value.
    7. Convert the updated list back to JSON text.
    8. Save it into SQLite.
    9. Commit and return success/failure.

Expected return:

    Recommended:

        {
            "ok": True,
            "message": "Entry updated successfully.",
            "operation": "add",
            "column": "synonym",
            "value": "glad"
        }

    Error example:

        {
            "ok": False,
            "error": "Add/delete operations are only allowed for synonym and translation."
        }

Use cases:

    Replace notes:

        update_entry_v2("English", "happy", "notes", "A common positive adjective.", "replace")

    Replace the whole synonym list:

        update_entry_v2("English", "happy", "synonym", "joyful, cheerful, pleased", "replace")

    Add one synonym:

        update_entry_v2("English", "happy", "synonym", "glad", "add")

    Delete one synonym:

        update_entry_v2("English", "happy", "synonym", "joyful", "delete")

Implementation notes:

    SQL placeholders cannot protect column names, so column must be checked with a whitelist before using f-string SQL.

    The synonym and translation fields are stored as TEXT in SQLite, but they represent JSON lists.

    This design is acceptable for the current MVP learning version. In the final graph-based version, synonym and translation relationships should eventually move to relations_v2.

2.11 delete_entry_v2(language, word)
------------------------------------------------------------

Function signature:

    delete_entry_v2(language, word)

What this function is:

    A CRUD Delete function. It removes one vocabulary entry from entries_v2.

What it is used for:

    When the user wants to delete a word from the notebook.

Parameters:

    language: str
        The language of the entry.

    word: str
        The word to delete.

How it works:

    1. Clean language.
    2. Normalize word.
    3. Find the entry id using get_entry_id().
    4. If the entry does not exist, return an error.
    5. Delete relations where this id is the source_entry_id or target_entry_id.
    6. Delete the entry itself.
    7. Commit the transaction.
    8. Use rowcount to confirm deletion.
    9. Return success result.

Expected return:

    Recommended:

        {
            "ok": True,
            "message": "Entry deleted successfully.",
            "relations_deleted": 2
        }

Use case:

    delete_entry_v2("English", "happy")

Implementation notes:

    There are two possible designs:

    Option A: Manual cleanup
        Delete relations first, then delete the entry.

    Option B: Foreign key cascade
        Define relations_v2 foreign keys with ON DELETE CASCADE.

    For learning clarity, manual cleanup is easier to understand.

    relation_exists(source_id, target_id, relation_type)

What this function is:

    A helper function that checks whether a relation already exists.

What it is used for:

    source_id: int
        The id of the source entry.

    target_id: int
        The id of the target entry.

    relation_type: str
        The type of relation, such as "synonym", "translation", "antonym", or "related".

Example:

    add_relation_v2() should call this before inserting a new relation.

Implementation notes:

    For symmetric relation types like synonym, the team needs to decide whether A -> B and B -> A are both stored, or whether list_relations_v2() handles both directions by query.

2.13 add_relation_v2(source_language, source_word, target_language, target_word, relation_type)
------------------------------------------------------------

Function signature:

    add_relation_v2(source_language, source_word, target_language, target_word, relation_type)

What this function is:

    A relation Create function. It creates a link between two vocabulary entries.

What it is used for:

    It allows the notebook to understand that two words are connected.

Parameters:

    source_language: str
        Language of the first/source word.

    source_word: str
        First/source word.

    target_language: str
        Language of the second/target word.

    target_word: str
        Second/target word.

    relation_type: str
        Type of relationship.

Suggested relation_type values:

    synonym
    translation
    antonym
    related
    example_of
    form_of

How it works:

    1. Normalize source_word and target_word.
    2. Clean source_language and target_language.
    3. Find source_entry_id using get_entry_id().
    4. Find target_entry_id using get_entry_id().
    5. If either entry does not exist, return an error.
    6. Validate relation_type.
    7. Check relation_exists(source_id, target_id, relation_type).
    8. If duplicate, return a duplicate relation result.
    9. Insert the relation into relations_v2.
    10. Commit the transaction.
    11. Return success result.

Expected return:

    Recommended:

        {
            "ok": True,
            "message": "Relation added successfully.",
            "relation_id": 5
        }

Use case:

    add_relation_v2("English", "happy", "Chinese", "开心", "translation")

Implementation notes:

    The function should not silently create missing entries unless the team intentionally chooses that design. For MVP, missing entries should return an error.

2.14 list_relations_v2(language, word)

Function signature:

    list_relations_v2(language, word)

What this function is:

    A relation Read function. It lists all relations connected to one entry.

What it is used for:

    When the user wants to see all words related to a given word.

Parameters:

    language: str
        The language of the word being searched.

    word: str
        The word being searched.

How it works:

    1. Normalize word.
    2. Clean language.
    3. Get the entry id using get_entry_id().
    4. If the entry does not exist, return an empty list or an error.
    5. Query relations_v2 where:
           source_entry_id = entry_id
       OR  target_entry_id = entry_id
    6. Join relations_v2 with entries_v2 to get the connected word details.
    7. Return a list of relation dictionaries.

Expected return:

    list[dict]

Example return:

    [
        {
            "source_language": "English",
            "source_word": "happy",
            "target_language": "Chinese",
            "target_word": "开心",
            "relation_type": "translation",
            "direction": "outgoing"
        },
        {
            "source_language": "English",
            "source_word": "joyful",
            "target_language": "English",
            "target_word": "happy",
            "relation_type": "synonym",
            "direction": "incoming"
        }
    ]

Use case:

    list_relations_v2("English", "happy")

Implementation notes:

    This function is one of the most important functions for the app. It turns the database from a simple word list into a connected vocabulary graph.

    It should ideally show both outgoing and incoming relations, because the user probably cares about the connection regardless of direction.

2.17 get_related_words_v2(language, word, relation_type=None)
------------------------------------------------------------

Function signature:

    get_related_words_v2(language, word, relation_type=None)

What this function is:

    A relation query function that returns connected words in a clean, UI-friendly format.

What it is used for:

    It gives the app an easy way to show synonyms, translations, antonyms, or other related words.

Parameters:

    language: str
        Language of the starting word.

    word: str
        Starting word.

    relation_type: str | None
        Optional filter. If provided, return only that relation type. If None, return all relation types.

How it works:

    1. Normalize word.
    2. Clean language.
    3. Get the entry id using get_entry_id().
    4. If entry does not exist, return an empty list or error.
    5. Query relations where this entry appears as source or target.
    6. If relation_type is provided, filter by it.
    7. For each relation, identify the other connected entry.
    8. Return only the related words in a clean list.

Expected return:

    list[dict]

Example return:

    [
        {
            "language": "English",
            "word": "joyful",
            "relation_type": "synonym"
        },
        {
            "language": "Chinese",
            "word": "开心",
            "relation_type": "translation"
        }
    ]

Use case:

    get_related_words_v2("English", "happy")
    get_related_words_v2("English", "happy", "synonym")

Difference from list_relations_v2():

    list_relations_v2() returns detailed relation information including source, target, and direction.

    get_related_words_v2() returns a simpler list of the connected words, which is easier for UI display and future AI features.

Implementation notes:

    This function is useful for the future vocabulary graph and AI recommendation system.

Parameters:

    language: str
        The language of the entry, such as "English" or "Chinese".

    word: str
        The vocabulary word, such as "happy" or "开心".

How it works:

    1. Clean the language.
    2. Normalize the word.
    3. Query entries_v2 for a row where language and word match.
    4. If the entry exists, return its id.
    5. If the entry does not exist, return None.

Expected return:

    int | None

Example:

    get_entry_id_v2("English", "happy") -> 3
    get_entry_id_v2("English", "unknownword") -> None

Use case:

    add_record_v2() needs entry_id before it can insert a new record into records_v2.

Implementation notes:

    This function should only find an existing entry.
    It should not create a new entry automatically.

    Returning None is useful because other functions can check whether the word exists before continuing.

    If the project already uses get_entry_id(language, word), the team should choose one naming style and use it consistently.
    For the current v2 function list, get_entry_id_v2(language, word) is the clearer name.

------------------------------------------------------------
2.19 add_record_v2(language, word, record_type, detail="")
------------------------------------------------------------

Function signature:

    add_record_v2(language, word, record_type, detail="")

What this function is:

    A Create function for the records_v2 table.
    It adds a learning, review, quiz, or note record for a vocabulary entry.

What it is used for:

    When the user learns, reviews, quizzes, or writes a note for a word, this function stores that action in the database.

Parameters:

    language: str
        The language of the vocabulary entry.

    word: str
        The vocabulary word.

    record_type: str
        The type of learning record.

        Suggested allowed values:
            learn
            review
            quiz_correct
            quiz_wrong
            note

    detail: str
        Optional extra information about the record.

        Example:
            "Reviewed this word today."
            "Got this word correct in quiz."
            "This word is used in positive emotion contexts."

How it works:

    1. Check whether record_type is valid.
    2. Use get_entry_id_v2(language, word) to find the entry id.
    3. If the entry does not exist, return an error or print a not-found message.
    4. Open a database connection.
    5. Insert a new row into records_v2 with entry_id, record_type, and detail.
    6. Let SQLite automatically create id and created_at.
    7. Commit the transaction.
    8. Close the connection.
    9. Return or print a success message.

Expected return:

    Recommended structured result:

        {
            "ok": True,
            "message": "Record added successfully.",
            "entry_id": 3,
            "record_type": "review"
        }

    Simple MVP version:

        None, with print messages.

Use case:

    add_record_v2("English", "happy", "learn", "First learned this word.")
    add_record_v2("English", "happy", "review", "Reviewed today.")
    add_record_v2("English", "happy", "quiz_correct", "Answered correctly.")

Implementation notes:

    This function depends on get_entry_id_v2().

    records_v2 should store entry_id instead of storing language and word again.
    This avoids duplicate data and keeps the record connected to the correct entry.

    The function should validate record_type so random values like "abc" are not stored.

2.20 list_records_v2(language=None, word=None)
------------------------------------------------------------

Function signature:

    list_records_v2(language=None, word=None)

What this function is:

    A Read function for the records_v2 table.
    It lists learning records stored in the database.

What it is used for:

    When the user wants to see learning history, review history, quiz history, or notes.

Parameters:

    language: str | None
        Optional. If provided without word, list records for that language.

    word: str | None
        Optional. If provided with language, list records for one specific word.

Supported behavior:

    list_records_v2()
        Lists all records.

    list_records_v2("English")
        Lists all records for English words.

    list_records_v2("English", "happy")
        Lists all records for the English word "happy".

How it works:

    1. Open a database connection.
    2. Join records_v2 with entries_v2 using entry_id.
    3. If both language and word are provided, filter by that exact entry.
    4. If only language is provided, filter by language.
    5. If neither is provided, return all records.
    6. Order records by created_at, usually newest first.
    7. Convert results into a list of dictionaries or print each record.
    8. Close the connection.
    9. Return the records.

Expected return:

    Recommended:

        list[dict]

Example return:

        [
            {
                "record_id": 1,
                "language": "English",
                "word": "happy",
                "record_type": "learn",
                "detail": "First learned this word.",
                "created_at": "2026-05-06 20:15:00"
            },
            {
                "record_id": 2,
                "language": "English",
                "word": "happy",
                "record_type": "review",
                "detail": "Reviewed today.",
                "created_at": "2026-05-06 20:20:00"
            }
        ]

Use case:

    list_records_v2()
    list_records_v2("English")
    list_records_v2("English", "happy")

Implementation notes:

    This function should use JOIN because records_v2 only stores entry_id.
    The JOIN allows the function to display language and word from entries_v2.

    This function is useful for future UI pages such as:
        learning history page
        word detail page
        review progress page

2.21 delete_record_v2(record_id)
------------------------------------------------------------

Function signature:

    delete_record_v2(record_id)

What this function is:

    A Delete function for the records_v2 table.
    It deletes one specific learning record by its record id.

What it is used for:

    When the user wants to remove an incorrect or unnecessary learning history record.

Parameters:

    record_id: int
        The id of the record in records_v2.

How it works:

    1. Open a database connection.
    2. Run DELETE on records_v2 where id equals record_id.
    3. Use cur.rowcount to check whether anything was deleted.
    4. If rowcount is 0, the record was not found.
    5. If rowcount is greater than 0, the record was deleted.
    6. Commit the transaction.
    7. Close the connection.
    8. Return or print the result.

Expected return:

    Recommended:

        {
            "ok": True,
            "message": "Record deleted successfully.",
            "record_id": 5
        }

    If record does not exist:

        {
            "ok": False,
            "error": "Record not found."
        }

Use case:

    delete_record_v2(5)

Implementation notes:

    This function deletes only the learning record.
    It should not delete the vocabulary entry itself.

    This is different from delete_entry_v2().
    delete_entry_v2() removes a word, while delete_record_v2() only removes one history item.

2.22 count_records_v2(language, word)
------------------------------------------------------------

Function signature:

    count_records_v2(language, word)

What this function is:

    A statistics/helper function that counts learning records for one vocabulary entry.

What it is used for:

    It helps the app understand how often a word has been learned, reviewed, answered correctly, answered wrongly, or noted.

Parameters:

    language: str
        The language of the vocabulary entry.

    word: str
        The vocabulary word.

How it works:

    1. Use get_entry_id_v2(language, word) to find the entry id.
    2. If the entry does not exist, return an error or empty result.
    3. Query records_v2 for all records connected to that entry_id.
    4. Group the records by record_type.
    5. Count how many records exist for each record_type.
    6. Return the result as a dictionary or list of dictionaries.

Expected return:

    Recommended:

        {
            "learn": 1,
            "review": 3,
            "quiz_correct": 2,
            "quiz_wrong": 1,
            "note": 2
        }

    Alternative list format:

        [
            {"record_type": "learn", "count": 1},
            {"record_type": "review", "count": 3},
            {"record_type": "quiz_correct", "count": 2}
        ]

Use case:

    count_records_v2("English", "happy")

Possible output:

    review: 5
    quiz_correct: 3
    quiz_wrong: 1
    note: 2

Implementation notes:

    This function is useful for future review and quiz features.

    Later, this function can help calculate:
        how often a word was reviewed
        whether the user often gets the word wrong
        which words need more practice
        basic spaced repetition priority

    For the MVP version, simple COUNT and GROUP BY are enough.

3. MVP IMPLEMENTATION ORDER
============================================================

Recommended build order:

1. get_connection()
2. create_tables()
3. normalize_word()
4. split_words()
5. entry_exists()
6. get_entry_id()
7. add_entry_v2()
8. find_entry_v2()
9. list_entries_v2()
10. update_entry_v2()
11. delete_entry_v2()
12. relation_exists()
13. add_relation_v2()
14. list_relations_v2()
15. delete_relation_v2()
16. search_entries_v2()
17. get_related_words_v2()
18. get_entry_id_v2()
19. add_record_v2()
20. list_records_v2()
21. delete_record_v2()
22. count_records_v2()

Reason for this order:

    Helper functions should be built before larger CRUD functions.
    Entry functions should be stable before relation functions.
    Relation functions require entry ids, so entries must work first.
    Search and related-word functions are easier after CRUD and relations are complete.
    Record functions should come after entries are stable, because each record must connect to an existing entry.

============================================================
5. FINAL PRODUCT MANAGER NOTES
============================================================

The goal for this version is not to build the AI system yet. The goal is to create a strong database foundation.

The database layer should support:

1. Clean vocabulary entry storage.
2. Safe duplicate checking.
3. Reliable update and delete behavior.
4. Word-to-word relations.
5. Simple search.
6. UI-friendly return values.

Once this version is stable, future work can add or improve:

1. records_v2 for learning history, review history, quiz history, and notes.
2. FTS5 for better full-text search.
3. fuzzy search for typo tolerance.
4. embeddings for AI semantic search.
5. graph traversal for multi-hop related words.
6. import/export for backup.

The most important engineering standard is this:

    database.py should be predictable, safe, and reusable.

It should not depend on terminal input, print-based UI, or hardcoded menu behavior. It should provide clean functions that can later be used by CLI, React, Electron, or API layers.
